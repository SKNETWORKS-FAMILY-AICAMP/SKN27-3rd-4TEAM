## 1. RAG 환각 방지 & 출처 표시

### 환각 방지

**시스템 프롬프트 규칙 (`backend/rag_server/core/llm.py`)**

```
"반드시 제공된 컨텍스트 문서에 근거하여 답변하고,
컨텍스트에 없는 내용은 추측하지 마세요."

"확실하지 않은 내용은 '제공된 근거만으로는 단정하기 어렵습니다'라고
말하고 전문가 상담을 안내하세요."
```

**리랭킹 기반 컨텍스트 품질 관리 (`backend/rag_server/core/rag_pipeline.py`)**

단순 벡터 유사도로 끝내지 않고 `_rerank_score()`로 2차 필터링:

| 조건 | 처리 |
|------|------|
| doc_type 일치 (법령/판례/사례집) | +0.12 ~ +0.22 부스팅 |
| must_terms 전체 매칭 | +0.08 |
| must_terms 하나도 미매칭 | -0.18 페널티 |
| "본문바로출력", "제목없음" 등 노이즈 제목 | -0.12 페널티 |
| 내용 120자 미만 짧은 청크 | -0.08 페널티 |
| 부칙·다른 법률의 개정 섹션 | -0.08 페널티 |

임계값 `MIN_FORCED_RERANK_SCORE = 0.12`, `FORCED_RERANK_SCORE_RATIO = 0.45`로 저품질 청크를 컷오프.

**계약서 진단 경로는 LLM 의존 없음**

`DiagnosisService` → `ModelAgent` + `SpecialTermsAgent`는 규칙 기반으로 위험 요소를 판단하므로 LLM 환각 문제가 원천 차단됨. LLM은 채팅(`LegalAgent`) 경로에서만 사용.

**평가 근거 데이터 (`rag/ragas_test.py`, `ragas_test_jeonse.py`)**

RAGAS 평가 스크립트 존재. Faithfulness / Answer Relevancy 수치를 측정·제시하면 발표 시 강력한 근거가 됨.

---

### 출처 표시

`RAGPipeline.chat()` → `_build_references()` → `RagReference` 객체:

```python
# 반환 필드
{
    "references": [
        {
            "file_name": "주택임대차보호법.pdf",
            "chunk_index": 12,
            "doc_type": "법령",
            "title": "주택임대차보호법 제3조"
        }
    ]
}
```

`ChatResponse`에 `references` 필드 포함, Neo4j 그래프 결과는 `graph_context`로 별도 반환.

> **주의**: 백엔드에서 references를 반환하지만, Streamlit 프론트(`views/chat.py`)에서 실제로 렌더링되는지 확인 필요. UI에 출처가 보이지 않으면 평가자 입장에서는 "없는 것"과 동일.

## 2. RDB & GraphDB 설계

### RDB (PostgreSQL + pgvector)

**테이블 구조**

| 테이블 | 역할 | 주요 컬럼 |
|--------|------|-----------|
| `rag_documents` | RAG 원본 청크 저장 | doc_type, title, file_name, chunk_index, chunk_text, vector_id |
| `langchain_pg_embedding` | pgvector 실제 벡터 | LangChain PGVector 자동 생성 |
| `diagnosis_logs` | 진단 결과 이력 | session_id, risk_score, risk_level, risk_factors(JSONB), rag_references(JSONB), estimated_sale_price, jeonse_ratio, contract_info(JSONB) |
| `sale_transactions` | 매매 실거래가 | 시군구, 법정동, dong_name, property_name, 거래금액, 건축연도 |
| `jeonse_transactions` | 전세 실거래가 | 시군구, 법정동, 보증금액, 건축연도 |

**설계 포인트**

- `risk_factors`, `rag_references`, `contract_info` → JSONB 저장으로 스키마 변경 없이 필드 확장 가능
- `ON CONFLICT DO NOTHING` → 중복 진단 로그 방지
- `vector_id IS NULL` 조건으로 임베딩 미완료 청크만 선별 재처리 (embed_to_pg.py)
- `ON CONFLICT ... DO UPDATE SET dong_name = EXCLUDED.dong_name WHERE ... IS NULL` → 기존 데이터 손상 없이 누락 컬럼만 보완

---

### GraphDB (Neo4j)

**노드 종류 (`rag/ingestion/build_graph.py`)**

| 노드 | 개수 | 설명 |
|------|------|------|
| `Domain` | 7개 | 계약, 등기, 보증, 법적절차, 시장분석, 특약검토, 사기피해 |
| `DocumentCategory` | 10개 | 판례, 법령, 서식, 사례집, 가이드, 보고서, 보증약관, 질의응답, 정책자료, 시세데이터 |
| `AgentScope` | 8개 | 특약검토, 등기·권리분석, 시장위험분석 등 LangGraph 에이전트별 담당 영역 |
| `LegalConcept` | 15개 | 대항력, 우선변제권, 확정일자, 전입신고, 임차권등기 등 |
| `Procedure` | 7개 | 전입신고, 확정일자, 임차권등기명령 등 임차인 절차 |
| `RiskFactor` | 12개 | RF001(전세가율) ~ RF012(계약갱신) |
| `Law` | - | 개별 법령 조문 |
| `Case` | - | 판례·사례집 항목 |

**주요 관계**

```
REGULATED_BY, EVIDENCED_BY, MITIGATED_BY, BELONGS_TO, DETECTED_BY,
CITES, CITED_IN, INVOLVES, DEMONSTRATES, REQUIRES, RELATED_TO,
DEFINED_IN, RESOLVES, GOVERNED_BY, USED_BY, COVERS, IN_CATEGORY
```

**실제 활용**

벡터 검색으로 잡기 어려운 관계 체인을 보완:

```
"대항력" → REQUIRES → "전입신고" → DEFINED_IN → "주택임대차보호법 제3조"
"근저당" → DETECTED_BY → RF002 → MITIGATED_BY → "임차권등기명령"
```

`graph_store.py`의 `get_risk_factors_by_keywords()`, `get_context_by_question()`에서 키워드 매칭으로 조회 → RAG 컨텍스트에 추가.

## 3. VectorDB 청킹 기준

### 문서 유형별 청킹 전략 (`rag/scripts/pdf_pipeline.py` v2)

| 유형 | 청크 크기 | 오버랩 | PDF 로더 | 선택 이유 |
|------|-----------|--------|----------|-----------|
| 판례 | 1,200자 | 150자 | `PyPDFLoader` | 판결문은 길고 문맥 연속성 중요 |
| 법령 | 1,000자 | 150자 | `PDFPlumberLoader` | 표·레이아웃 보존 필요 |
| 사례집·가이드 | 1,000자 | 150자 | `PyMuPDFLoader` | LangChain 0.3 호환 |

jm 경로 기본값 (`rag/jm/core/config.py`): `chunk_size=900`, `chunk_overlap=150` (환경변수 조정 가능)

**v1 → v2 개선 이유**

- 청크 크기 상향 (800→1000~1200): 법률 조문이 길어 800자 제한 시 조문 의미가 반쪽으로 잘리는 문제 발생
- 오버랩 확대 (80~100 → 150): 조문 경계에서 앞 문맥이 끊기는 문제 해소
- separator 정교화: `"\n제"` → `"\n제\d+조"` 패턴으로 조문 중간 분리 방지

---

### 2단계 청크 전처리 파이프라인

```
PDF 파일
  ↓ [1단계] pdf_pipeline.py — 관련성 필터 + 섹션 제거 + 청크 분할
  ↓ [2단계] clean_chunks.py — 노이즈 정제
  ↓ embed_to_pg.py — 임베딩 + pgvector 적재
```

**2단계 정제 (`rag/ingestion/clean_chunks.py`) 제거 대상**

| 제거 대상 | 예시 |
|-----------|------|
| HTML 태그 | `<br>`, `<td>`, `<b>` |
| 마크다운 기호 | `###`, `**bold**`, `---` |
| 표 구분선 | `\|---\|---\|` 형태 |
| 표 행 → 자연어 변환 | 셀 중복 제거 + pipe 제거 |
| 페이지 번호 줄 | 숫자 단독 행 |
| 사법정보공개포털 잔해 | "본문바로출력", breadcrumb 등 |
| 의미 없는 청크 | 한글+영문 20자 미만 → DB에서 삭제 |

이 정제 과정이 없으면 `| 제1조 | | | |` 같은 표 잔해가 임베딩되어 검색 품질이 크게 저하됨.

## 4. 데이터 수집 & 전처리

### 수집 데이터

| 데이터 | 출처 | 저장 위치 | 용도 |
|--------|------|-----------|------|
| 판례 PDF | 대법원 사법정보공개포털 | pgvector + Neo4j | 법적 근거 검색 |
| 법령 PDF | 주택임대차보호법, 민법, 공인중개사법 등 | pgvector + Neo4j | 조문 근거 검색 |
| 사례집 PDF | 국토부, 소비자원 전세사기 피해 사례 | pgvector | 유사 사례 검색 |
| 매매 실거래가 CSV | 국토부 실거래가 공개시스템 | PostgreSQL `sale_transactions` | 예상 매매가 산출 |
| 전세 실거래가 CSV | 국토부 실거래가 공개시스템 | PostgreSQL `jeonse_transactions` | 전세가율 산출 |

---

### PDF 전처리 흐름

**관련성 필터 (`pdf_pipeline.py`)**

```python
CORE_KEYWORDS = ["전세", "임대차", "임차인", "보증금 반환", ...]
NOISE_KEYWORDS = ["폭행", "횡령", "마약", "강도", "이혼", ...]

# 핵심 키워드 0개 → 스킵
# 노이즈 키워드 2개 이상 + 핵심 1개 이하 → 스킵
```

대법원 판례는 형사·가사 사건이 대량 혼재하므로 이 필터로 불필요한 문서를 적재 전에 제거.

**판례 섹션 필터링**

```python
KEEP_SECTIONS = ["이유", "주문", "참조조문", "참조판례"]
# 【당사자】, 【소송대리인】 등 법적 내용 없는 섹션 제거
```

**판례 메타데이터 추출 (파일명 + 텍스트 파싱)**

```python
{
    "court": "서울고등법원",          # 법원명
    "decision_date": "2023-05-12",   # 선고일
    "case_no": "2022나12345",         # 사건번호
    "keywords": ["전세사기", "대항력"] # 핵심 키워드 태깅
}
```

---

### 실거래가 CSV 전처리 (`rag/ingestion/load_market_data.py`)

**NaN → None 변환 (int 오버플로 방지)**

```python
# pandas float64 NaN을 그대로 int 변환하면 대형 음수값 발생
# → astype(object).where(notna()) 패턴으로 Python None으로 변환
result = result.astype(object).where(result.notna(), other=None)
```

**중복 처리 전략**

```sql
ON CONFLICT (시군구, 법정동, 거래금액, 계약일)
DO UPDATE SET dong_name = EXCLUDED.dong_name
WHERE sale_transactions.dong_name IS NULL
```

기존 레코드 덮어쓰기 없이 누락된 `dong_name` 컬럼만 보완.
