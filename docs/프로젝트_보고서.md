## 1. RAG 환각(Hallucination) 방지 및 출처 표시

### 1-1. 환각 방지 전략

본 시스템은 **"LLM이 판례를 지어내지 못하도록"** 검색→인용→응답의 3단 파이프라인을 강제합니다.

**(A) 검색 우선(Retrieval-First) 구조 — `backend/agents/legal_agent.py`**

```
사용자 질문
   ↓ (1) LLM이 쟁점 키워드만 추출   ← JSON 배열 강제
   ↓ (2) Neo4j 그래프에서 실제 판례 검색
   ↓ (3) 검색된 판례 텍스트를 컨텍스트로 LLM에 전달
   ↓ (4) "근거 자료 없음 → 그렇게 답변" 규칙
최종 답변 (사건번호·법조문 인용)
```

핵심은 **LLM은 검색만 못 하게** 한 것입니다. 1단계에서 LLM은 사전 정의된 25개 키워드(보증금반환, 대항력, 우선변제권, 근저당, 전세권, 임차권등기, 확정일자, 사기죄, 배임, 경매, 신탁 등) 중에서만 골라내도록 프롬프트가 제약됩니다. 추출된 키워드로 Neo4j가 실제 저장된 85개 판례 중에서 매칭되는 케이스를 반환하고, LLM은 그 결과 텍스트만 보고 답변합니다.

**(B) 프롬프트 레벨 가드레일**

- `LEGAL_PROMPT` (legal_agent.py:114): "관련 판례의 사건번호와 판결 요지를 구체적으로 인용하세요 / 확실하지 않은 사항은 전문가 상담을 권고하세요 / 법률 자문이 아닌 정보 제공임을 명시하세요"
- `temperature=0.0` (쟁점 추출), `temperature=0.3` (응답 생성) — 창의성 최소화
- 검색 결과 0건일 때는 `"관련 판례를 찾지 못했습니다."` 메시지 그대로 컨텍스트에 삽입 → LLM이 임의 판례를 생성할 여지 차단

**(C) 두 단계 분류로 RAG 트리거 격리 — `backend/agents/chatbot.py`**

`classify_question()`가 질문을 `legal` / `general`로 먼저 분류하여, 법률 질문일 때만 RAG 경로(`LEGAL_ENHANCED_PROMPT`)로 전환합니다. 일반 상담 질문은 진단 데이터 컨텍스트만 사용해 RAG 오남용을 막습니다.

**(D) 구조화 출력 강제**

모든 에이전트 LLM 호출은 Pydantic 모델 + JSON 형식 강제(`반드시 아래 JSON 형식으로만 응답하세요. 다른 텍스트는 포함하지 마세요.`)로 받습니다. 파싱 실패 시 자동 폴백(`parse_case_info_from_filename()` 등 정규식 백업) 처리. — `extract_entities.py:113`, `contract_extractor.py:114`

## 2. RDB · GraphDB 설계

### 2-1. PostgreSQL (pgvector) — RDB 역할

`docker-compose.yml`에 `pgvector/pgvector:pg16` 컨테이너가 정의되어 있으며, 다음 두 가지 용도로 사용됩니다.

| 용도 | 내용 |
|---|---|
| 실거래 데이터 저장 (선택) | 종로구 매매·전세 CSV의 정규화 저장소. 현재 구현은 CSV → pandas 직접 로딩(`data_loader.py`) 방식이며, 확장 시 pg에 적재 가능 |
| 진단 리포트 영속 (현재는 JSON) | `reports/report_<uuid>.json` 형태 파일 저장 — pg로 마이그레이션 가능한 구조 |
| 벡터 임베딩 (예정) | `sentence-transformers` + `pgvector` 조합으로 판례 본문 임베딩 저장용 인프라 준비 |

**연결 정보**: `postgresql://postgres:jeonse1234@localhost:5432/jeonse_risk`

### 2-2. Neo4j GraphDB — 판례 지식 그래프

**스키마 (graph_builder.py)**

```
Nodes:
  (:Case  {case_id, court, date, summary, filename})
  (:Law   {name})         ← 예: "주택임대차보호법 제3조"
  (:Issue {name})         ← 예: "보증금반환", "대항력"

Relationships:
  (Case)-[:CITES_LAW]->(Law)      ← 판례가 인용한 법조문
  (Case)-[:CITES_CASE]->(Case)    ← 판례가 인용한 다른 판례
  (Case)-[:DEALS_WITH]->(Issue)   ← 판례가 다루는 쟁점

Constraints:
  CREATE CONSTRAINT FOR (c:Case)  REQUIRE c.case_id IS UNIQUE
  CREATE CONSTRAINT FOR (l:Law)   REQUIRE l.name IS UNIQUE
  CREATE CONSTRAINT FOR (i:Issue) REQUIRE i.name IS UNIQUE
```

**그래프 통계 (현재 빌드 기준)**
- Case 노드: 85개 (대법원/고등법원/헌법재판소 판례 PDF)
- Issue 노드: 약 25개 (보증금반환, 대항력, 우선변제권, 근저당, 사기죄 등)
- Law 노드: 주택임대차보호법, 민법, 부동산등기법 등 인용 법조문
- 평균 fan-out: 판례 1건당 법조문 2~5개 + 쟁점 2~4개

**주요 쿼리** — `query_by_issue()` (graph_builder.py:106)

```cypher
MATCH (i:Issue) WHERE i.name CONTAINS $keyword
MATCH (c:Case)-[:DEALS_WITH]->(i)
OPTIONAL MATCH (c)-[:CITES_LAW]->(l:Law)
OPTIONAL MATCH (c)-[:CITES_CASE]->(cited:Case)
RETURN c.case_id, c.court, c.date, c.summary,
       collect(DISTINCT l.name) AS laws,
       collect(DISTINCT cited.case_id) AS cited_cases
ORDER BY c.date DESC
```

`query_related_cases(case_id, depth=2)`는 가변 깊이 경로 탐색(`-[*1..2]-`)으로 연관 판례 네트워크를 추출합니다.

## 3. VectorDB 구축 — 문서 청킹 기준

### 3-1. 청킹 전략 (`backend/graph/extract_entities.py`)

본 프로젝트는 일반적인 토큰 기반 sliding-window 청킹이 아니라, **"판례 1건 = 1 청크"의 엔티티 단위 청킹**을 채택했습니다.

| 항목 | 값 | 근거 |
|---|---|---|
| 청킹 단위 | 1 PDF = 1 Case 엔티티 | 판례는 사건번호 단위로 독립된 논리 단위 |
| 최대 입력 길이 | 2,000자 (한글) | `text[:2000]` (extract_entities.py:104) — GPT-4o-mini 비용/지연 절충 |
| 메타데이터 추출 | case_id, court, date, summary, cited_laws, cited_cases, issues | LLM 구조화 출력 |
| 폴백 청크 | LLM 실패 시 파일명 정규식 파싱 | `parse_case_info_from_filename()` |
| 임베딩 대상 | summary + issues (그래프 노드) | 의미 검색은 Neo4j `CONTAINS` 매칭으로 대체 |
| pgvector 준비 | requirements.txt에 `pgvector>=0.3.0`, `sentence-transformers>=3.0.0` 명시 (확장 슬롯) |

### 3-2. 청킹 이유

1. **법률 문서 특성**: 판례는 사건번호 단위로 의미가 닫혀있어 임의 분할 시 쟁점이 잘릴 위험이 큼
2. **검색 정밀도 우선**: 쟁점 키워드 정확 매칭이 의미 임베딩 유사도보다 신뢰도 높음 — 그래프 직접 쿼리 채택
3. **인용 추적**: 판례→판례 인용 관계(`CITES_CASE`)를 보존하려면 사건번호 단위 청크가 필수

### 3-3. 임베딩 확장 설계

vector 검색이 필요해질 경우의 확장 경로:

```sql
CREATE TABLE case_embeddings (
  case_id    TEXT PRIMARY KEY,
  summary    TEXT,
  embedding  VECTOR(384)      -- sentence-transformers/paraphrase-MiniLM
);
CREATE INDEX ON case_embeddings USING ivfflat (embedding vector_cosine_ops);
```

| 데이터 | 출처 | 건수 | 기간 |

### 4-2. 실거래 CSV 컬럼 구조

```
연립다세대 매매: house_name, dong_name, sgg_name, jibun, exclusive_area_m2,
                deal_amount, floor, build_year, deal_date
오피스텔   전세: officetel_name, dong_name, ..., deposit_amount, monthly_rent, ...
```

### 4-3. 전처리 파이프라인 (`backend/data_loader.py`)

```
1. load_all_csv()
   ├─ 40개 CSV → pandas concat
   ├─ 파일명 prefix(연도) 파싱 → year 컬럼 추가
   ├─ officetel_name / house_name → building_name으로 통일
   ├─ housing_type = "오피스텔" | "연립다세대"
   └─ 전세 데이터는 monthly_rent == 0 필터 (순수 전세만)

2. area_bucket() — 면적 구간화
   ├─ ~33㎡       (원룸급)
   ├─ 33~66㎡    (소형)
   ├─ 66~99㎡    (중형)
   └─ 99㎡~       (대형)

3. generate_predictions() — 2027 예측
   ├─ (dong_name, area_bucket, year) 그룹별 평균 deposit / sale 산출
   ├─ CAGR 계산: (last_val/first_val)^(1/n) - 1
   ├─ 2025 기준 × (1+CAGR)^2 → 2027 예측
   ├─ 전세가율 = 예측전세금 / 예측매매가 × 100
   └─ 위험도: ≥80% 위험 / ≥70% 주의 / 그외 안전
```

### 4-4. 판례 PDF 전처리 (`backend/graph/extract_entities.py`)

```
PDF → PyMuPDF로 본문 추출 → 2000자 truncate → LLM JSON 추출
                                              ↓
파일명 정규식 폴백:                ┌──────────────────────────┐
  법원: "대법원|헌법재판소|...고법" │ case_id  : "YYYYNNN가형식"│
  날짜: r'(\d{4})\.\s*(\d{1,2})...'│ court    : 법원명           │
                                  │ date     : YYYY-MM-DD       │
                                  │ summary  : 2~3문장           │
                                  │ cited_laws / cited_cases    │
                                  │ issues                       │
                                  └──────────────────────────┘
                                              ↓
                                       Neo4j MERGE 삽입
```

### 4-5. 결측·이상치 처리

- 동(dong) 데이터 부족(2건 미만): 예측 스킵 (`if len(group) < 2: continue`)
- 매매가 결측: `predicted_sale_2027=None`, `jeonse_ratio=None`, 위험도="미상"
- LLM 추출 실패: 파일명 정규식 폴백 + `summary="LLM 추출 실패 - 파일명에서 기본 정보만 추출"`

### 7-3. 성능 지표

| 항목 | 측정값 | 비고 |
|---|---|---|
| 계약서 진단 1건 소요 시간 | 약 25~35초 | 특약 N건일 때 N×10초 |
| 그래프 빌드 (85 PDF 전체) | 약 8~12분 | LLM 추출이 병목 |
| Neo4j 쿼리 응답 | <50ms | 키워드 CONTAINS + 1-hop |
| 챗봇 응답 (법률) | 약 5~8초 | RAG 검색 + LLM 생성 |
| 챗봇 응답 (일반) | 약 2~3초 | LLM 단일 호출 |

### 7-4. 발견된 한계 및 개선점

- **한계 1**: 종로구 단일 자치구 한정 → 강남/마포 등 확장 시 동명 충돌 가능 (예: "역삼동" vs "역삼1동")
- **한계 2**: 면적 구간이 4단계로 거칠어, 30~33㎡ 경계의 작은 차이가 다른 버킷으로 분리
- **한계 3**: 월세 포함 계약서 차단 → MVP 범위 결정. 향후 보증부월세도 지원 예정
- **한계 4**: 판례 본문이 2,000자로 truncate되어 긴 판례의 후반부 쟁점 누락 가능
- **개선 방향**: pgvector 임베딩 도입 시 쟁점 키워드 미정의 질문도 의미 유사도 매칭 가능
