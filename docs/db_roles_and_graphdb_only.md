# DB 역할 정의 및 GraphDB 단독 사용 시 예상 결과

---

## 1. 현재 구조 — 두 DB가 하는 일

```
계약서 업로드
      │
      ▼
[ContractParser] ── 텍스트·특약사항 추출
      │
      ▼
[DiagnosisService]
      │
      ├─── ① Neo4j (GraphDB) ─────────────────────────────────────────┐
      │         "이 계약서에 어떤 위험 유형이 해당되는가?"              │
      │         키워드로 사전 정의된 위험 요소(RF001~RF010) 조회       │
      │         → 구조화된 위험 요소 목록 반환                         │
      │                                                               │
      └─── ② pgvector (VectorDB) ────────────────────────────────────┤
                "이 계약의 위험과 관련된 실제 법령·판례가 뭔가?"        │
                문장 의미 기반으로 적재된 문서 청크 검색               │
                → 관련 법령·사례집 내용 반환                           │
                                                                      │
                                    ▼                                 │
                          [LLM (GPT-4o-mini)] ◄────────────────────────┘
                          위험 요소 DB + 실제 문서 컨텍스트를
                          함께 받아서 risk_score·risk_factors·summary 생성
```

---

## 2. Neo4j (GraphDB) — 위험 유형 분류기

### 무엇을 저장하는가
미리 정의해둔 **위험 요소 지식 그래프**

| 노드 | 내용 |
|------|------|
| `RiskFactor` | RF001~RF010 — 위험 유형, 심각도(HIGH/MEDIUM/LOW), 대응 조언 |
| `Law` | 주택임대차보호법 제○조, 민법 제○조 등 |
| `Case` | 관련 판례명 |
| 관계 | `(RiskFactor)-[:REGULATED_BY]->(Law)` |

### 어떻게 쓰이는가
계약서에서 추출된 **키워드**("근저당", "전세가율", "특약" 등)로 Neo4j에 질의해서
해당 키워드가 포함된 `RiskFactor` 노드들을 가져온다.

```
입력: ["근저당", "전세가율", "전세보증보험"]
  │
  ▼ Neo4j Cypher 쿼리
출력: RF002(근저당·가압류 — HIGH), RF001(전세가율 80% — HIGH), RF010(보험미가입 — MEDIUM)
```

### 한마디로
> **"이 계약서에서 어떤 위험 패턴이 감지됐는가"를 사전 DB 기반으로 분류하는 역할**
> 항상 일관된 factor_id·category·severity·advice를 제공하는 **정형화된 위험 분류 엔진**

## 3. pgvector (VectorDB) — 법령·사례 참고 문서 검색기

### 무엇을 저장하는가
실제 **문서 청크**를 임베딩(숫자 벡터)으로 변환해 저장

| 문서 종류 | 예시 |
|-----------|------|
| 법령 | 주택임대차보호법 조문, 민법 조문 |
| 사례집 | 전세사기 피해예방 종합안내서 |
| 판례 | 전세피해 법률상담 사례집 |

### 어떻게 쓰이는가
"전세계약 위험 근저당 전세가율" 같은 검색 쿼리를 **의미 기반**으로 유사도 검색해서
가장 관련 있는 문서 청크 5개를 가져온다.

```
검색 쿼리: "전세계약 위험 근저당 전세가율"
  │
  ▼ 코사인 유사도 검색
출력: "전세사기 피해예방 안내서 - 선순위 권리관계 확인..." (score: 0.50)
      "전세피해법률상담사례집 - 배당순위 소액보증금..." (score: 0.497)
      ... 총 5개
```

### 한마디로
> **"이 위험과 관련된 실제 법령·사례가 무엇인가"를 문서에서 찾아주는 역할**
> 키워드가 아닌 문장의 **의미**로 검색하므로 다양한 표현도 잡아냄
> API 응답의 `references` 필드 = 이 검색 결과

---

## 4. 두 DB의 역할 비교 요약

| 구분 | Neo4j (GraphDB) | pgvector (VectorDB) |
|------|-----------------|---------------------|
| 저장 내용 | 위험 유형 지식 그래프 (구조화) | 실제 문서 텍스트 청크 (비정형) |
| 검색 방식 | 키워드 매칭 (Cypher 쿼리) | 의미 유사도 (코사인 거리) |
| 출력 결과 | risk_factors (RF001~RF010) | references (법령·사례 원문) |
| LLM 역할 | 위험 요소 템플릿 제공 | 근거 문서 컨텍스트 제공 |
| 없으면 | 위험 분류가 LLM에만 의존 → 일관성 저하 | 근거 없는 진단 → 신뢰도 저하 |

## 5. GraphDB(Neo4j)만 사용한다면?

### 달라지는 코드 흐름
```
기존: VectorDB 검색 → context 문서  +  Neo4j 검색 → risk_factors
변경: context = "관련 문서를 찾지 못했습니다."  +  Neo4j 검색 → risk_factors
```

`rag_pipeline.py`의 `diagnose()`에서 `_format_context(raw_results)`가 빈 결과를 받으면 고정 문자열 `"관련 문서를 찾지 못했습니다."`를 LLM에 넘긴다.

### 예상 응답 변화

**① references → 빈 배열**
```json
"references": []
```
근거 문서를 보여줄 수 없으므로 사용자 신뢰도 하락

**② risk_factors — 큰 변화 없음 (Neo4j가 살아있으므로)**
```json
"risk_factors": [
  { "factor_id": "RF002", "category": "권리관계", "severity": "HIGH", ... },
  { "factor_id": "RF001", "category": "전세가율",  "severity": "HIGH", ... },
  ...
]
```
Neo4j의 키워드 매칭은 그대로 동작하므로 위험 분류 자체는 유지된다.

**③ risk_score — 불안정해질 가능성 있음**

LLM이 context 문서 없이 점수를 산출해야 하므로, 법령·판례에 근거한 정교한 판단 대신
자체 학습 데이터에만 의존하게 된다. 결과적으로 점수가 실제보다 낮거나 높게 나올 수 있다.

**④ summary — 근거 없는 일반론**
```
기존: "계약서에는 근저당권 설정과 관련된 위험 요소가 존재하며...
       (전세사기 피해예방 종합안내서 참조)"

변경: "계약서에 근저당권이 있어 주의가 필요합니다.
       전문가와 상담하시기 바랍니다."
```
법령·사례집 원문 없이 생성되므로 요약이 피상적·일반적으로 변한다.

### GraphDB 단독 사용 시 예상 결과 예시
```json
{
  "risk_score": 75,
  "risk_level": "주의",
  "risk_factors": [
    { "factor_id": "RF002", "severity": "HIGH", "description": "근저당·가압류 과다..." },
    { "factor_id": "RF001", "severity": "HIGH", "description": "전세가율 80% 초과..." },
    { "factor_id": "RF010", "severity": "MEDIUM", "description": "전세보증보험 미가입..." }
  ],
  "summary": "근저당권 및 전세가율 관련 위험이 감지되었습니다. 전문가와 상담하시기 바랍니다.",
  "references": []
}
```

현재 결과(risk_score 85, references 5개)와 비교하면:
- risk_score 낮아지거나 불규칙 (법령 근거 없이 산출)
- summary 부실 (구체적 법령 인용 불가)
- references 없음 (사용자가 스스로 확인할 근거 부재)

---

## 6. 결론

> 이 시스템에서 두 DB는 **서로 보완**하는 구조다.
>
> - **Neo4j** 없으면: 위험 유형 분류가 일관되지 않고 LLM 응답마다 달라진다
> - **pgvector** 없으면: 법령·판례 근거가 없어 진단의 신뢰성이 떨어진다
>
> 전세계약 **위험 진단** 도메인에서 두 DB가 각자의 역할로 최적화되어 있으므로,
> 가능하면 둘 다 유지하는 것이 권장된다.
